## 체크리스트 목적: "요구사항 문서의 단위 테스트"

**핵심 개념**: 체크리스트는 구현 테스트가 아니라, 요구사항 문서의 품질을 검증하는 도구입니다.

**다음은 아님**
- ❌ 버튼 클릭 동작 검증
- ❌ 에러 처리 동작 테스트
- ❌ API 200 응답 확인
- ❌ 코드가 명세와 같은지 확인

**다음은 맞음**
- ✅ 모든 카드 유형의 시각 계층 요구가 정의되어 있는가?
- ✅ "눈에 띄게"가 크기/위치로 정량화되어 있는가?
- ✅ hover 요구가 상호작용 요소 전반에 일관적인가?
- ✅ 키보드 탐색 접근성 요구가 정의되어 있는가?
- ✅ 로고 이미지 로드 실패 시 동작이 정의되어 있는가?

## 사용자 입력

```text
$ARGUMENTS
```

입력이 비어 있지 않다면 반드시 고려해야 합니다.

## 실행 단계

1. **준비**: 루트에서 `.specify/scripts/bash/check-prerequisites.sh --json` 실행 후
   `FEATURE_DIR`, `AVAILABLE_DOCS` 파싱(모든 경로는 절대경로).

2. **의도 명확화(동적)**: 최대 3개의 초기 질문(Q1~Q3)을 생성합니다.
   - 사용자 문구 + spec/plan/tasks 신호 기반
   - 체크리스트 내용에 실질 영향이 있는 질문만
   - 이미 명확한 내용은 질문 생략
   - 정밀도를 우선

   생성 규칙:
   - 도메인 키워드, 리스크 신호, 이해관계자 힌트, 산출물 키워드 추출
   - 후보 포커스(최대 4개)로 군집화 후 우선순위
   - 대상 사용자/타이밍 추정(작성자, 리뷰어, QA, 릴리스)
   - 누락 차원 탐지(범위, 깊이, 리스크 강조, 제외 범위, 측정 기준)

   질문 형식:
   - 옵션형이면 `Option | Candidate | Why It Matters` 표 사용
   - A~E 이내
   - 이미 말한 내용 재질문 금지
   - 불확실하면 "X가 범위에 포함되는지 확인"처럼 명시 질문

   상호작용 불가 시 기본값:
   - 깊이: Standard
   - 대상: 코드 관련이면 Reviewer(PR), 아니면 Author
   - 포커스: 상위 2개 클러스터

   필요 시 Q4/Q5(최대 2개) 후속 질문 가능(사유 1줄 첨부).

3. **요청 해석**: `$ARGUMENTS` + 답변을 결합해 아래를 확정합니다.
   - 체크리스트 테마(보안/리뷰/배포/UX 등)
   - 사용자 필수 항목
   - 포커스 기반 카테고리 뼈대
   - 부족 정보는 spec/plan/tasks에서 보완(추정 생성 금지)

4. **기능 컨텍스트 로드** (`FEATURE_DIR`)
   - `spec.md`: 요구사항/범위
   - `plan.md`(있으면): 기술 맥락
   - `tasks.md`(있으면): 구현 작업

   로딩 전략:
   - 활성 포커스와 관련 부분만 로드
   - 긴 본문은 요약 중심
   - 공백이 있으면 점진적으로 추가 로드

5. **체크리스트 생성**
   - `FEATURE_DIR/checklists/`가 없으면 생성
   - 파일명은 짧고 목적이 드러나게: `[domain].md` (예: `ux.md`, `api.md`)
   - 파일이 없으면 CHK001부터 생성
   - 파일이 있으면 마지막 CHK 번호 다음부터 append
   - 기존 내용 삭제/교체 금지

   **원칙: 구현이 아니라 요구사항 자체를 검증**
   - 완전성
   - 명확성
   - 일관성
   - 측정 가능성
   - 시나리오 커버리지

   **권장 카테고리**
   - Requirement Completeness
   - Requirement Clarity
   - Requirement Consistency
   - Acceptance Criteria Quality
   - Scenario Coverage
   - Edge Case Coverage
   - Non-Functional Requirements
   - Dependencies & Assumptions
   - Ambiguities & Conflicts

   **항목 작성 규칙**
   - 질문형 문장
   - 문서에 "써 있는 것"을 검증
   - 품질 차원 태그 포함: `[Completeness]` 등
   - 가능하면 `[Spec §X.Y]` 참조 포함
   - 누락 점검은 `[Gap]` 사용

   **금지 패턴(구현 테스트화 금지)**
   - "Verify/Test/Confirm" + 동작 검증
   - 클릭/렌더/실행 같은 구현 행위 표현
   - QA 절차/테스트케이스 중심 문구

   **필수 패턴(요구사항 품질 검증)**
   - "[시나리오]에 대한 요구사항이 정의되어 있는가?"
   - "모호한 용어가 정량화되어 있는가?"
   - "섹션 간 요구사항이 충돌 없이 일관적인가?"
   - "요구사항이 객관적으로 검증 가능한가?"

   **추적성 규칙**
   - 최소 80% 항목에 참조/표식 포함
   - `[Spec §X.Y]`, `[Gap]`, `[Ambiguity]`, `[Conflict]`, `[Assumption]`

   **항목 수 관리**
   - 후보가 40개 초과면 위험/영향 기준으로 축약
   - 유사항목 통합

6. **구조 준수**
   - `.specify/templates/checklist-template.md` 구조를 따릅니다.
   - 템플릿이 없으면 아래 최소 구조 사용:
     - H1 제목
     - 목적/생성일 메타
     - `##` 카테고리
     - `- [ ] CHK### ...` 항목

7. **결과 보고**
   - 체크리스트 파일 절대경로
   - 항목 개수
   - 신규 생성인지 append인지
   - 선택된 포커스/깊이/사용 주체
   - 사용자 지정 필수 항목 반영 여부

## 참고

각 `/speckit.checklist` 실행은 목적별 파일을 생성 또는 append하여
`checklists/` 폴더에서 유형별(예: `ux.md`, `security.md`)로 관리합니다.
