## 사용자 입력

```text
$ARGUMENTS
```

입력이 비어 있지 않다면 반드시 고려해야 합니다.

## 목표

구현 전에 3개 핵심 산출물(`spec.md`, `plan.md`, `tasks.md`) 사이의 불일치, 중복,
모호성, 미정의를 식별합니다. 이 명령은 `/speckit.tasks`로 완전한 `tasks.md`가
생성된 뒤에만 실행해야 합니다.

## 운영 제약

**엄격한 읽기 전용**: 어떤 파일도 수정하지 않습니다. 구조화된 분석 리포트만 출력합니다.
필요 시 개선안(수정 계획)을 제안할 수 있지만, 실제 편집은 사용자 명시 승인 후 별도 실행입니다.

**헌장 우선 원칙**: `.specify/memory/constitution.md`는 이 분석 범위에서 절대 기준입니다.
헌장 MUST 위반은 자동으로 CRITICAL입니다. 해결은 spec/plan/tasks 조정으로 해야 하며,
원칙 약화·재해석·묵시는 허용되지 않습니다.

## 실행 단계

### 1. 분석 컨텍스트 초기화

저장소 루트에서 한 번 실행:

`.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks`

JSON에서 `FEATURE_DIR`, `AVAILABLE_DOCS`를 파싱하고 절대경로를 구성합니다.

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

필수 파일이 없으면 중단하고 선행 명령 실행을 안내합니다.

### 2. 산출물 로드(점진적 공개)

최소한의 필요한 부분만 읽습니다.

**spec.md**
- 개요/문맥
- 기능 요구사항
- 비기능 요구사항
- 사용자 스토리
- 엣지 케이스(있다면)

**plan.md**
- 아키텍처/스택 선택
- 데이터 모델 참조
- 단계(Phase)
- 기술 제약

**tasks.md**
- Task ID
- 설명
- Phase 그룹
- 병렬 표식 `[P]`
- 참조 파일 경로

**헌장**
- `.specify/memory/constitution.md` 원칙 검증

### 3. 의미 모델 구성

출력에는 원문 전체를 포함하지 말고 내부 모델만 구성합니다.

- **요구사항 목록**: 기능/비기능 요구사항 + 안정 키
- **사용자 행동 목록**: 수용 기준이 붙은 사용자 액션
- **Task 매핑**: 각 Task를 요구사항/스토리에 매핑
- **헌장 규칙셋**: 원칙명과 MUST/SHOULD 문장 추출

### 4. 탐지 패스

고신호 이슈 중심으로 최대 50개까지만 보고하고, 나머지는 요약합니다.

#### A. 중복 탐지
- 유사/중복 요구사항 탐지
- 더 명확한 문구로 통합 권고

#### B. 모호성 탐지
- 측정 불가 형용사(빠른, 안전한, 직관적인 등) 탐지
- TODO/TKTK/???/placeholder 미해결 표식 탐지

#### C. 미정의 탐지
- 동사는 있으나 목적/측정 결과가 없는 요구사항
- 수용 기준 정렬이 약한 사용자 스토리
- spec/plan에 없는 컴포넌트를 참조하는 task

#### D. 헌장 정합성
- 헌장 MUST 충돌 항목
- 헌장 필수 섹션/게이트 누락

#### E. 커버리지 공백
- task가 0개인 요구사항
- 어떤 요구사항/스토리에도 매핑되지 않은 task
- task에 반영되지 않은 비기능 요구사항

#### F. 불일치
- 용어 표류(같은 개념의 다른 명칭)
- plan/spec 엔터티 불일치
- task 순서 모순(의존성 설명 없이 선후관계 역전)
- 상충 요구사항(예: Next.js vs Vue)

### 5. 심각도 분류

- **CRITICAL**: 헌장 MUST 위반, 핵심 산출물 누락, 핵심 요구사항 무커버리지
- **HIGH**: 충돌/중복 요구사항, 보안·성능 모호성, 검증 불가 수용 기준
- **MEDIUM**: 용어 표류, 비기능 task 누락, 엣지 케이스 미정의
- **LOW**: 표현 개선, 실행 순서에 영향 없는 경미한 중복

### 6. 리포트 출력(파일 쓰기 금지)

다음 구조의 Markdown 리포트를 출력합니다.

## 명세 분석 리포트

| ID | 범주 | 심각도 | 위치 | 요약 | 권고 |
|----|------|--------|------|------|------|
| A1 | Duplication | HIGH | spec.md:L120-134 | ... | ... |

행당 1개 이슈, 안정 ID 사용.

**커버리지 요약 표**

| Requirement Key | Has Task? | Task IDs | Notes |
|-----------------|-----------|----------|-------|

**헌장 정합성 이슈** (있다면)

**미매핑 Task** (있다면)

**지표**
- Total Requirements
- Total Tasks
- Coverage %
- Ambiguity Count
- Duplication Count
- Critical Issues Count

### 7. 다음 액션 제안

- CRITICAL이 있으면 `/speckit.implement` 전에 해결 권고
- LOW/MEDIUM만 있으면 진행 가능하되 개선 제안 제공
- 예시 명령 제안:
  - `/speckit.specify` 재정제
  - `/speckit.plan` 아키텍처 조정
  - `tasks.md` 수동 편집으로 공백 보완

### 8. 개선안 제안 여부 질문

마지막에 묻습니다:

"상위 N개 이슈에 대한 구체 수정안을 제안할까요?"

(자동 적용 금지)

## 운영 원칙

### 컨텍스트 효율

- 고신호 이슈 중심
- 점진적 로딩
- 출력 최대 50개
- 동일 입력 시 결정론적 결과

### 분석 가이드

- 파일 수정 금지
- 없는 내용을 추정 생성 금지
- 헌장 위반 최우선
- 일반론보다 구체 사례 인용
- 이슈 0건이면 통계와 함께 정상 보고
